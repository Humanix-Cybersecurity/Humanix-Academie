debrief: |
  Le **offboarding** d'un prestataire est l'etape la plus negligee de
  la relation. Le contrat est termine, l'attention se relache, et des
  acces techniques restent ouverts pendant des mois ou des annees -
  c'est la source de nombreux breach par "compte fantome".

  **La checklist exhaustive en 15 points :**

  **Acces utilisateurs (jour J)**

  **1. Comptes nominatifs individuels**
  - Disable AD/Azure/SSO pour chaque user du prestataire
  - Verifier que la connexion est effectivement bloquee (test)
  - Revoquer les tokens d'application persists (Office 365 OAuth
    tokens, MFA app passwords)

  **2. Comptes partages "maintenance" / "support"**
  - Renommer ET changer le mot de passe immediatement
  - Si plus utiles : desactiver puis supprimer apres 30 jours
  - **Ne JAMAIS partager le nouveau mot de passe avec qui que ce
    soit qui pourrait etre lie au prestataire**

  **3. Acces VPN**
  - Revoquer les certificats client VPN
  - Disable les comptes VPN nominatifs
  - Auditer les logs VPN des 90 derniers jours pour detecter activite
    anormale

  **4. SSH / acces serveur**
  - Supprimer les cles SSH du prestataire des `~/.ssh/authorized_keys`
  - Audit `auth.log` pour detecter dernieres connexions
  - Si compte partage type `maintenance` : changer le mot de passe
    racine immediatement

  **Acces API et integrations (jour J + 1)**

  **5. Tokens API**
  - Lister tous les tokens generes pour le prestataire (app
    integrations, webhooks, API keys)
  - Revoquer chaque token
  - Generer de nouveaux tokens pour les integrations qui doivent
    persister (et utiliser un nouveau compte service)

  **6. Certificats clients TLS / mTLS**
  - Revoquer les certificats client (CRL ou OCSP)
  - Verifier que les API qui utilisaient ces certs refusent les
    connexions

  **7. Integrations tierces (Slack, Teams, Jira, Confluence)**
  - Auditer les apps OAuth installees par le prestataire
  - Revoquer les integrations
  - Auditer les guests/external members et les retirer

  **Donnees (jour J + 7)**

  **8. Donnees stockees chez le prestataire**
  - Demander preuve de suppression (cf. clause contractuelle ep 04)
  - Si extraction de production a ete faite chez eux pour test :
    demander confirmation de suppression de cette copie
  - Recuperer la version finale des donnees au format export ouvert

  **9. Documentation technique chez le prestataire**
  - Recuperer copies de Confluence/Notion/SharePoint
  - Demander suppression du Confluence apres recuperation
  - Si docs sensibles : audit des acces logs pour voir qui a
    consulte quoi

  **10. Backups & archives**
  - Demander preuve de suppression des backups (souvent oublies)
  - Politique de retention : si le prestataire doit conserver 1 an
    pour conformite legale, en conserver les certificats de
    destruction differe

  **Infrastructure (jour J + 7)**

  **11. DNS records**
  - Si le prestataire avait des sous-domaines deleges (ex.
    `monitoring.acme.fr` -> NS chez eux) : reprendre le controle
  - Verifier les enregistrements MX/SPF/DKIM (s'il envoyait des mails
    depuis ton domaine)

  **12. Comptes service & integrations applicatives**
  - Lister tous les comptes service utilises par le prestataire
    (CI/CD pipelines, deployment, monitoring)
  - Generer nouveaux credentials, mettre a jour les pipelines

  **13. Webhooks sortants & abonnements**
  - Liste des webhooks que TU envoies vers le prestataire (Slack/
    Teams notifications, etc.)
  - Desactiver

  **Aspects organisationnels (jour J + 14)**

  **14. Documentation & connaissance**
  - Recuperer les knowledge bases, scripts, runbooks crees par le
    prestataire
  - Faire un transfert de competences avec leur equipe avant depart
  - Documentation interne pour le successeur / equipe

  **15. RGPD - donnees personnelles employes prestataire**
  - Si tu avais en BDD les noms/email des employes du prestataire
    (annuaire, gestion d'acces) : verifier la politique de
    retention
  - Anonymiser/supprimer apres la fin du contrat selon ta politique
    de retention

  **Le piege "On garde une copie au cas ou"**

  Tu peux etre tente de garder l'acces du prestataire "au cas ou
  on aurait besoin de lui rappeler quelque chose pendant 1-2
  semaines". MAUVAISE IDEE.

  Solution : si vraiment besoin, garde un compte temporaire avec
  expiration **automatique** (pas "on coupera quand on aura le
  temps"), MFA obligatoire, et logs surveilles. Mais l'experience
  montre que tu n'auras pas besoin de lui acceder.

  **Audit final (jour J + 30) :**

  Apres 30 jours, faire un audit complet :
  - Aucune connexion VPN du prestataire dans les logs
  - Aucun acces SSH avec leurs credentials
  - Aucun token API actif
  - Aucun email entrant/sortant avec leurs comptes
  - Documentation interne complete (le tien et eux n'ont plus rien)

  Si le moindre acces actif persiste : action immediate.

  **Cas reel - Tesla 2018 :**

  Un ancien sous-traitant Tesla, dont le contrat avait pris fin
  6 mois plus tot, a utilise des credentials non revoques pour se
  connecter au reseau interne et exfiltrer 300 Go de donnees
  sensibles (code, plans). L'incident a ete decouvert plusieurs
  semaines apres l'exfiltration.

  Cause : oubli de revocation lors de l'offboarding. Defense :
  checklist exhaustive systematique, audit periodique des comptes
  fantomes.
