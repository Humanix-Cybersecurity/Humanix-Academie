debrief: |
  Donner un acces a un prestataire est l'une des situations les plus
  risquees en cybersecurite. Le breach de Target 2013 (110M cartes
  bancaires volees) a commence par un acces VPN d'un fournisseur de
  climatisation mal securise.

  **Les 8 controles a appliquer (la grille complete) :**

  **1. Compte dedie + nominatif**

  - **NON** : compte partage `migration` utilise par 5 personnes du
    prestataire.
  - **OUI** : un compte par individu (`alice.dupont@prestataire.fr` =
    compte `alice-dupont-presta`). Si le prestataire envoie 5
    consultants, 5 comptes.

  Avantage : tu peux revoquer/auditer par individu.

  **2. MFA obligatoire**

  - TOTP (Google Authenticator, Authy) : minimum acceptable.
  - **WebAuthn / FIDO2** : ideal (cle physique YubiKey, ou passkey).
  - SMS : a eviter (interceptable par SIM swapping).

  Sans MFA = compte compromettable par phishing/brute-force. Refuser.

  **3. VPN segmente (pas SSH direct sur internet)**

  Ton prestataire doit se connecter a TON VPN d'entreprise (Cisco
  AnyConnect, OpenVPN, WireGuard) qui le place dans un VLAN dedie
  "Prestataires" avec acces UNIQUEMENT a la machine cible (serveur
  ERP) sur les ports necessaires (SSH 22, HTTPS 443).

  Pas d'acces lateral aux autres machines.

  **4. Jump server / bastion**

  Pour les acces critiques, le prestataire ne se connecte pas
  directement au serveur cible. Il passe par un **bastion** :
  - Le bastion est un serveur dedie (Linux minimal)
  - L'acces SSH du prestataire est sur le bastion uniquement
  - Le bastion proxy les connexions vers les serveurs cibles
  - Le bastion **enregistre toutes les sessions** (asciinema,
    teleport, gravitational)

  Avantage : tu peux rejouer la session du prestataire si tu doutes
  de ce qu'il a fait. Forensic facile.

  Solutions : Teleport (gratuit OSS), Hashicorp Boundary, AWS Session
  Manager, ou Wallix Bastion (FR commercial).

  **5. Acces just-in-time (just-enough access)**

  - **Just-in-time** : le compte n'est actif que pendant les
    fenetres prevues (ex : du lundi au vendredi 9h-18h, hors weekend).
  - **Just-enough** : le compte n'a que les droits minimum
    necessaires pour la tache (pas root global, juste les binaires
    necessaires via sudo whitelist).

  Outils : Teleport, BeyondTrust, CyberArk PAM.

  **6. Logs centralises et audites**

  Toutes les actions du prestataire doivent etre logguees :
  - Connexion VPN (heure, IP source, succes/echec)
  - Sessions SSH (qui, quoi, quand, depuis ou)
  - Commandes executees (auditd Linux, ou shell history)
  - Fichiers transferes (SCP/SFTP)

  Logs centralises dans un SIEM (Wazuh, Elastic, Sekoia FR) pour
  analyse anti-anomalie.

  **7. Expiration automatique**

  Le compte du prestataire DOIT avoir une date d'expiration
  configurable :
  - **Linux** : `usermod -e 2024-12-31 username`
  - **Active Directory** : Account expires
  - **VPN** : tag d'expiration sur le compte

  Apres la date, le compte est automatiquement disable. Pas
  d'oublis qui creent des "comptes fantomes" actifs des annees apres.

  **8. Notification & approbation**

  Pour les acces les plus sensibles :
  - **Notification** : email automatique a 2 personnes en CC quand
    le prestataire se connecte. Detection de connexions hors-fenetre.
  - **Approbation** : pour acces a la prod, le prestataire doit
    demander un OTP a un humain qui l'approuve. Reduit le risque en
    cas de compte compromis.

  **Le piege "VPN d'entreprise = acces a tout"**

  Beaucoup d'entreprises ont un VPN unique qui donne acces a tout le
  reseau (full network). Le prestataire qui s'y connecte peut scanner
  / acceder a TOUTE l'infra.

  Solution : VPN avec **micro-segmentation** ou **ZTNA** (Zero Trust
  Network Access) - chaque user/role a des regles d'acces specifiques
  basees sur la cible, pas un acces full network.

  Solutions ZTNA : Cloudflare Access, Tailscale (OSS), Twingate, ou
  bien le NAC en interne (FreeRADIUS + 802.1X).

  **Cas Target 2013 (rappel) :**

  Le fournisseur de climatisation Fazio Mechanical avait un compte
  VPN avec acces au reseau Target pour faire de la facturation
  electronique. Pas de segmentation : depuis ce VPN, acces a TOUT le
  reseau Target, y compris les caisses des magasins. Les attaquants
  ont compromis Fazio (phishing), recupere les credentials VPN, et
  pivote vers les caisses. 110M de cartes bancaires volees.

  Si Target avait segmente le VPN au seul service de facturation
  (not lateral movement possible), l'attaque s'arretait la.

  **Pour les developpeurs :**

  Si TU es le prestataire qui se connecte a un client :
  - Insiste pour avoir un compte nominatif (pas un compte partage)
  - Demande l'MFA des le premier jour
  - Documente toutes tes actions (commande, heure)
  - Si le client n'a aucun de ces controles : signale-le a ton
    manager, c'est un risque pour toi (en cas de breach, tu seras
    accuse meme si tu n'es pour rien).
