debrief: |
  Les pipelines CI/CD sont des **cibles de choix** pour les attaquants :
  ils contiennent les secrets les plus critiques (cles cloud, NPM tokens,
  GPG keys, deploy keys SSH), tournent souvent en mode automatique, et
  beaucoup de devs y appliquent moins de rigueur que sur le code prod.

  C'est OWASP **A05:2021 Security Misconfiguration** + **A02:2021
  Cryptographic Failures** + **supply chain (A08)**.

  **Les 5 risques majeurs sur un pipeline CI/CD :**

  **1. PR de fork externe avec secrets exposes**

  Sur GitHub Actions, par defaut depuis 2021, les workflows sur PR de
  forks externes **ne recoivent PAS les secrets**. Mais :
  - Si tu utilises `pull_request_target:` (au lieu de `pull_request:`),
    les secrets SONT envoyes (et c'est volontairement risque).
  - Si tu approuves manuellement le run d'un PR externe, les secrets
    arrivent.

  Defense :
  - **NE JAMAIS** utiliser `pull_request_target` sans comprendre ce que
    ca implique.
  - Configurer Settings > Actions > "Approval required" pour les PR
    externes.
  - Lire le diff du PR AVANT d'approuver le run - chercher des
    modifications de `.github/workflows/`, `Dockerfile`, scripts shell.

  **2. Cles long-lived au lieu d'OIDC**

  Anti-pattern :
  ```yaml
  - uses: aws-actions/configure-aws-credentials@v4
    with:
      aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
      aws-secret-access-key: ${{ secrets.AWS_SECRET_KEY }}
  ```

  Bon pattern (OIDC) :
  ```yaml
  permissions:
    id-token: write
    contents: read
  steps:
    - uses: aws-actions/configure-aws-credentials@v4
      with:
        role-to-assume: arn:aws:iam::123:role/github-deploy
        aws-region: eu-west-3
  ```

  OIDC = pas de secret long-lived stocke. Le token est genere a chaque
  run (~1h de validite), scope a un seul role IAM. Si le pipeline est
  compromis, l'attaquant a au max 1h de Compromise window vs un secret
  long-lived qui peut etre vole et utilise pendant des mois.

  AWS, GCP, Azure, HashiCorp Vault, npm (provenance), tous supportent
  OIDC avec GitHub Actions et GitLab CI depuis 2022-2023.

  **3. Actions tierces non pinned (supply chain)**

  Anti-pattern :
  ```yaml
  - uses: some-org/some-action@main  # toujours la derniere version
  ```

  L'attaquant qui compromet le repo `some-org/some-action` peut
  injecter du code malicieux qui s'executera dans TES runs.

  Bon pattern :
  ```yaml
  - uses: some-org/some-action@a4b5c6d7e8f9...  # SHA pinne
  # OU
  - uses: actions/checkout@v4   # tag officiel d'une org trusted
  ```

  Pin les SHA pour les actions de tiers de confiance moyenne.
  Tag uniquement pour les actions officielles GitHub (`actions/*`)
  ou de tres grosses orgs (Docker, AWS, Azure officiel).

  Outil : `dependabot` peut updater les SHA automatiquement.

  **4. Logs qui leak les secrets**

  ```yaml
  - run: echo "Token is $MY_TOKEN"  # MAUVAIS si MY_TOKEN n'est pas
                                     # un secret GitHub (pas masque)
  - run: echo "Token is ${{ secrets.MY_TOKEN }}"
                          # GitHub masque AUTO les secrets dans les logs
                          # MAIS uniquement si la valeur exacte apparait
  ```

  Pieges :
  - Si tu base64 ou modifies le secret, le masking ne reconnait plus
    la valeur.
  - Si tu fais `echo "$MY_TOKEN" | head -c 5` (pour debug), le start
    du secret apparait en clair.

  Defense :
  - Utiliser `::add-mask::` GitHub action si tu as des secrets calcules.
  - Setup `set -x` est un PIEGE en CI - ca log toutes les commandes
    avec leurs args, secrets compris.

  **5. Scope IAM trop large**

  Le role IAM utilise par le pipeline doit avoir le **minimum** de
  permissions :
  - Production deploy : juste les S3 buckets / EC2 / ECS de prod, pas
    "AdministratorAccess".
  - Tests : pas d'acces prod, pas d'IAM admin.
  - Build : pas d'acces aux secrets manager autres que ceux du build.

  Reviewer regulierement les policies IAM. Outil : `iam-floyd` (TS) ou
  AWS IAM Access Analyzer.

  **Bonus - GitLab CI vs GitHub Actions :**

  GitLab CI a aussi son OIDC (`id_tokens:`) supporte depuis 2023.
  GitLab a en plus :
  - **Protected branches/tags** : variables exposees seulement sur ces
    refs. **A activer pour les secrets prod.**
  - **Masked variables** : log redaction (equiv GitHub).
  - **CI/CD variable scope** : par projet / par groupe / par instance.

  **Pour ton workflow GitHub d'aujourd'hui, check ces 5 cases :**

  - [ ] OIDC partout ou ca marche (AWS/GCP/Azure/npm)
  - [ ] Actions tierces pin SHA (sauf actions/* officielles)
  - [ ] Pas de `pull_request_target` sauf necessite + audit
  - [ ] Settings > Actions > Approval required pour les PR externes
  - [ ] Audit des roles IAM utilises - principle of least privilege
