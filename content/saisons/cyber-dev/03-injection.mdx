debrief: |
  L'injection est **OWASP A03:2021** - reste dans le top 3 mondial des
  vulnerabilites web depuis 20 ans. C'est la faille la plus emblematique
  du dev backend.

  **Le mecanisme commun aux 3 types d'injection :**

  Le code construit une commande (SQL, shell, LDAP query, ...) en
  **concatenant** une chaine venant de l'utilisateur. L'attaquant
  envoie une chaine qui change la SEMANTIQUE de la commande au lieu
  d'être traitee comme donnee.

  **Type 1 - SQL Injection (la plus connue)**

  ```typescript
  // VULNERABLE - concatenation SQL
  const q = `SELECT * FROM users WHERE id = ${req.params.id}`;
  // payload : ?id=1 OR 1=1
  // SQL execute : SELECT * FROM users WHERE id = 1 OR 1=1
  // -> retourne TOUS les users
  ```

  Defense canonique : **prepared statements / parametrized queries**.
  Tous les ORM/drivers serieux les supportent :
  - **Prisma** : `db.user.findUnique({ where: { id } })`
  - **node-postgres** : `pool.query("SELECT * FROM users WHERE id = $1", [id])`
  - **Knex** : `knex("users").where("id", id)`
  - **Java JDBC** : `PreparedStatement` avec `?` placeholders

  Ne JAMAIS concatener du SQL. Même si "c'est juste un id numérique" -
  un id peut être un string, un objet, un payload encode.

  **Type 2 - Command Injection (la plus destructrice)**

  ```typescript
  // VULNERABLE - exec shell
  exec(`ping -c 1 ${req.body.host}`);
  // payload : "host": "google.com; rm -rf /"
  // shell execute : ping -c 1 google.com; rm -rf /
  ```

  Defense canonique :
  1. **N'execute jamais une commande shell construite a partir d'input
     utilisateur**. Si possible, utilise un binding API (genre `dns.lookup()`
     pour ping), pas un `exec()`.
  2. Si vraiment necessaire : `execFile(cmd, [args])` qui passe les
     arguments comme tableau (le shell ne les interprete pas).
  3. Whitelist stricte sur les inputs (regex `^[a-z0-9.-]+$` pour un
     hostname).

  **Type 3 - LDAP Injection (la plus discrete)**

  ```javascript
  // VULNERABLE - filter LDAP concatene
  const filter = `(&(uid=${req.body.username})(userPassword=${req.body.password}))`;
  // payload username : "*)(uid=*"
  // filter execute : (&(uid=*)(uid=*)(userPassword=...))
  // -> matche TOUS les utilisateurs
  ```

  Defense canonique :
  - Utilise un client LDAP qui escape les caracteres spéciaux
    (`(`, `)`, `*`, `\`, `\\u0000`).
  - **ldapjs** en Node fait l'escape via `Filter` builders.
  - Authentification : NE FAIS PAS de bind avec password concatene -
    utilise le bind LDAP comme test (success/fail) plutôt qu'extraire
    le password en clair.

  **Les autres injections a connaître :**

  - **NoSQL Injection** : MongoDB avec `$where`, payload `{"$gt": ""}`.
  - **XPath Injection** : si tu fais des queries XPath dans du XML.
  - **Template Injection (SSTI)** : Jinja, Handlebars, Pug qui prennent
    user input dans le template = RCE potentielle.
  - **HTTP header injection** : injection de `\r\n` dans les redirects.

  **La règle d'or universelle :**

  > **Separer DONNEES et CODE.**
  >
  > Les données passent par des paramètres bind (prepared statements,
  > placeholders, args arrays). Jamais par concatenation textuelle.

  **Outils pour détecter avant la prod :**

  - **CodeQL** (GitHub, gratuit OSS) : detecte la majorité des injections
    a la PR.
  - **Semgrep** (gratuit) : règles SAST custom.
  - **Bearer** (FR, gratuit OSS) : focus sécurité + privacy.
  - **OWASP ZAP** : scanner runtime.
  - **Linters ORM-aware** : `@typescript-eslint/no-unsafe-call` + règles
    custom contre `$queryRawUnsafe`.

  **Tests en code review :**

  Tu vois une concatenation de string qui finit dans une `query()`,
  `exec()`, ou un filter LDAP avec un input utilisateur ? **Stop, demande
  une parametrized version.** C'est la règle la plus simple et la plus
  efficace.
