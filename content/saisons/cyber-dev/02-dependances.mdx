debrief: |
  La supply chain JS est l'**OWASP A06:2021 - Vulnerable and Outdated
  Components** ET **A08:2021 - Software and Data Integrity Failures**
  combines. C'est l'une des surfaces d'attaque les plus exploitees en
  2024-2026.

  **Les 3 risques majeurs sur tes dependances :**

  **1. CVE non patches (vulnerable components)**
  - Une lib que tu utilises a une faille publique connue (CVE+CVSS).
  - Outil : `npm audit`, `pnpm audit`, `yarn audit`, ou GitHub Dependabot.
  - Reflexe : update vers la version patchee (souvent un patch ou minor
    suffit). Les `--force` qui font des majors sont a manier avec soin.

  **2. Typosquatting (supply chain)**
  - Un attaquant publie un package avec un nom proche d'un legitime :
    `colourful-logs` (UK) vs `colorful-logs` (US), `react-native-caméra`
    vs `react-native-cam`, etc.
  - Cas reel : `eslint-scope@3.7.2` (octobre 2018) compromis avec un
    payload qui volait les npm tokens. Affecte 4500 packages downstream.
  - Détection : audit manuel des nouveaux packages au PR review +
    outils comme `socket.dev` (gratuit) qui flagge les packages suspects.

  **3. Dependency confusion**
  - L'attaquant publie sur npm public un package avec le même nom qu'un
    package interne de ton entreprise. Si ton tooling resout les
    dependances dans le mauvais ordre, il peut tirer le package public
    malicieux.
  - Cas reel : Birsan 2021, breach chez Apple, Microsoft, Tesla, Yelp.
  - Defense : namespace tes packages internes (`@acme/utils` au lieu de
    `acme-utils`), configure ton `.npmrc` avec `registry=` interne.

  **L'outillage moderne pour gerer ça :**

  **GitHub Dependabot** (gratuit, integre)
  - PR automatiques pour les CVE patchables.
  - Alertes par email quand un nouveau CVE touche ton projet.
  - Active dans Settings > Code security and analysis.

  **Renovate** (open source, plus puissant)
  - Comme Dependabot mais configurable a fond (règles de groupage,
    cadences différentes par scope, automerge sur patches uniquement).
  - YAML config dans `renovate.json` a la racine.
  - Recommande pour les projets serieux > 50 dependences.

  **Socket.dev** (gratuit, flagge supply chain)
  - Detecte typosquatting, install scripts suspects, telemetrie
    cachee, dependances fraiches d'un mainteneur inconnu.
  - PR comments inline qui expliquent ce qui est suspect.

  **`npm-audit-resolver`** ou **`audit-ci`**
  - Permet de whitelister des CVE acceptes (ex: faux positifs ou
    ReDoS dans une dep buildtime non exposee).

  **La règle d'or 2026 :**

  Sur un projet en prod, en moyenne 60 a 80% de tes dependances sont
  des transitives (deps de tes deps). Tu ne peux pas les auditer toutes
  manuellement. La defense est **automatisee + en couches** :

  1. CI bloque sur high/critical CVE non patches
  2. Dependabot/Renovate proposent les fixes
  3. Socket.dev flagge les supply chain risks
  4. Code review humaine sur les NEW deps (jamais sur les bumps)
  5. Lock file commit (package-lock.json, yarn.lock, pnpm-lock.yaml)
  6. Audit semestriel des deps obsoletes a degager

  **Pour les projets souverains français :** privilegier les deps
  maintenues par des fondations (Apache, Linux Foundation, OpenJS) ou
  des entreprises FR/EU. Éviter les packages mainteneurs solo non
  identifiables.
