debrief: |
  La regle absolue : **un secret pushe sur GitHub est un secret COMPROMIS**.

  Peu importe que le repo soit public ou prive (les insiders, les fork
  sauvages, les leak de tokens... tout arrive). Peu importe que tu effaces
  l'historique : les bots scrapper de cles (truffleHog, gitleaks, Github's
  own secret scanner) rament les nouveaux commits en quelques dizaines de
  secondes et stockent les matchs.

  **Procedure d'urgence quand tu detectes un commit avec secret :**

  1. **REVOQUE LE SECRET** en premier (cle API : dashboard de l'editeur ;
     mot de passe : changement immediat ; certificat : revocation CA).
     C'est la SEULE action qui rende le secret inutilisable. Avant de
     toucher au code.

  2. **Genere un nouveau secret** et update les deploiements (vault,
     secrets manager, CI/CD vars).

  3. **Audit des logs** d'usage du secret revoque pendant la fenetre
     d'exposition (Stripe / AWS / GitHub te montrent l'historique des
     calls). Si requetes suspectes : escalade incident.

  4. **Nettoie l'historique Git** (avec `git filter-repo` ou BFG, plus
     `git push --force-with-lease`). Important pour la propete du repo,
     mais pas la priorite securite.

  5. **Post-mortem** : pourquoi le secret est arrive la ? Pre-commit
     hook manquant ? Code review qui a laisse passer ? Workflow brise ?

  **Defense en amont (l'ordre des priorites) :**

  **Niveau 1 - Outils developpeur (gratuit, facile)**
  - **`gitleaks`** ou **`detect-secrets`** en pre-commit hook : bloque le
    commit si secret detecte. Setup 5 min via Husky ou pre-commit.com.
  - **GitHub Secret Scanning** : active dans Settings > Security. Detecte
    50+ types de secrets (AWS, Stripe, Slack tokens, etc.) sur push.

  **Niveau 2 - Architecture (le bon investissement)**
  - **Aucun secret dans le code, jamais.** Variables d'environnement
    chargees depuis un vault au runtime.
  - **Vault souverain** : HashiCorp Vault, Doppler, Bitwarden Secrets,
    AWS Secrets Manager, Azure Key Vault, GCP Secret Manager.
  - **`.env.example` versionne, `.env` jamais.** Le fichier `.env` doit
    etre dans `.gitignore` et chaque dev a sa version locale.

  **Niveau 3 - Politique d'equipe**
  - **Code review obligatoire** sur les PRs qui touchent aux secrets/
    configs (les seniors ont le reflexe, les juniors apprennent).
  - **Rotation reguliere** des secrets (90 jours pour les criticals).
  - **Principe de moindre privilege** : la cle Stripe utilisee en dev
    n'est PAS la cle live - utilise des cles de test.

  **Le piege classique :** "j'ai juste commit un secret pour debugger,
  je l'enleverai apres" - non. Tu cree un debt securite immediat.
  Utilise une variable d'env meme pour 30 secondes de debug.

  **Pour l'open source :** si tu maintiens un projet OSS, **active**
  GitHub secret scanning push protection (Settings > Secret scanning >
  Push protection ON). Ca refuse le push si un secret est detecte, avec
  override possible pour les faux positifs.

  Cas reel 2024 : un dev a commit une cle AWS dans un repo public. En
  47 minutes, des bots ont mine 90 000 dollars de Bitcoin sur EC2 avant
  qu'il ne s'en rende compte. La revocation aurait pris 10 secondes.
