debrief: |
  La code review sécurité n'est pas un audit complet. C'est une grille
  de **questions canoniques** que tu deroules sur chaque PR backend.
  Avec de l'entrainement, ça devient instinctif (~5 min par PR).

  **La grille en 7 points (a memoriser) :**

  **1. Authentification ET autorisation**

  ```typescript
  // INSUFFISANT
  const session = await auth();
  if (!session?.user) return 401;

  // CORRECT
  const session = await auth();
  if (!session?.user) return 401;
  if (session.user.rôle !== "ADMIN") return 403;
  // ET tenantId scope
  const where = { tenantId: session.user.tenantId };
  ```

  Auth ≠ Autz. **Auth** = qui es-tu ? **Autz** = qu'as-tu le droit de
  faire ?

  Reflexe : sur tout endpoint qui retourne des données, demande
  "qui peut acceder a quoi" en parametrant par rôle + tenantId/orgId.

  **2. Inputs trustes vs untrustes**

  Tout ce qui vient du client est **untruste** :
  - `req.body`, `req.query`, `req.params`
  - Headers (sauf `Authorization` valide côté serveur)
  - Cookies (sauf signed/HttpOnly/Secure verifies)
  - Données venant d'autres services qui ne sont pas authentifies

  La session NextAuth est **trustee** (signee par AUTH_SECRET) - utilise
  `session.user.tenantId` pas `req.query.tenantId`.

  Reflexe : "d'ou vient cette donnee ?" - si réponse = "client", elle
  doit être validee (Zod schéma) ET autorisee (rôle/scope check).

  **3. Validation d'input (Zod / Joi / Yup)**

  ```typescript
  // BON
  const Schema = z.object({
    name: z.string().min(1).max(100),
    email: z.string().email(),
    rôle: z.enum(["LEARNER", "MANAGER"]),
  });
  const parsed = Schema.safeParse(await req.json());
  if (!parsed.success) return 400;
  ```

  Reflexe : tout endpoint qui prend du body JSON DOIT avoir un
  schéma strict. Pas de `as any`, pas de "je suppose que c'est
  bien forme".

  **4. SELECT explicite des champs**

  ```typescript
  // DANGEREUX - retourne tous les champs incluant secrets
  const user = await db.user.findUnique({ where: { id } });

  // BON - allowlist explicite
  const user = await db.user.findUnique({
    where: { id },
    select: { id: true, email: true, name: true },
    // PAS : passwordHash, mfaSecret, lockedUntil, ipHash, ...
  });
  ```

  Reflexe : sur chaque retour API, "est-ce que ce champ peut sortir ?"
  Specialement : passwordHash, mfaSecret, sessionToken, apiKey,
  resetToken, lockedUntil, ipHash, persona prive, etc.

  **5. Pas de PII dans les logs**

  ```typescript
  // BAD
  console.log(`User ${user.email} logged in from ${ip}`);

  // BETTER
  console.log(`User ${user.id} logged in from ${hashIp(ip)}`);
  ```

  RGPD article 5 : limitation. Les logs sont conserves longtemps,
  partages avec les outils de monitoring (Datadog, Sentry, etc.) -
  pas le bon endroit pour des emails ou des IPs en clair.

  Reflexe : si tu vois `user.email`, `req.body`, ou `req.headers`
  dans un `console.log` - flagge.

  **6. Opérations destructives = double check**

  ```typescript
  // DANGEREUX
  await db.user.delete({ where: { id: req.params.id } });

  // BON
  await db.user.delete({
    where: {
      id: req.params.id,
      tenantId: session.user.tenantId, // empeche cross-tenant
    },
  });
  // + vérifier qui peut delete (rôle)
  // + log audit (tracabilite RGPD article 30)
  ```

  Pour `delete`, `update` de rôles, `transfer` financier : auth + rôle
  + scope + audit log obligatoires.

  **7. Erreurs : ne fuite pas le SQL/stack**

  ```typescript
  // BAD - leak des structures internes
  catch (e) {
    return NextResponse.json({ error: e.message, stack: e.stack }, { status: 500 });
  }

  // BON
  catch (e) {
    console.error("[users/export]", e); // log serveur, pas client
    return NextResponse.json({ error: "internal_error" }, { status: 500 });
  }
  ```

  **La culture en code review :**

  - **Ne dis pas "c'est faux"**. Dis "qu'est-ce qui empeche que cet
    endpoint soit appele par un user d'un autre tenant ?"
  - **Lien vers la doc OWASP** plutôt que de re-expliquer (cheat sheet
    OWASP est gold standard).
  - **Suggere la fix** : pas juste "il faut un rôle check" mais
    "ajoute `if (session.user.rôle !== 'ADMIN') return 403`".
  - **Bloque le merge** sur les bugs critiques (auth, secrets, RCE).
    Approve avec commentaires sur les bugs mineurs (logs, naming).

  **Ce qui n'est PAS de la code review sécurité :**

  - Performance (autre review)
  - Style/lint (le linter le fait)
  - Tests (autre review)

  Reste focus sur les vulnerabilites. ~5 min par PR backend, ça evite
  des incidents qui couteraient des semaines de remediation.
