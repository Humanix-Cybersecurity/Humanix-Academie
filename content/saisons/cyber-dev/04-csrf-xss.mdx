debrief: |
  XSS et CSRF sont les deux faces complémentaires des attaques web :
  XSS exploite la confiance du browser dans le serveur, CSRF exploite
  la confiance du serveur dans le browser. Les deux sont OWASP **A03
  + A05**.

  **XSS - Cross-Site Scripting**

  L'attaquant injecte du JavaScript qui s'execute dans le navigateur
  de la victime, dans le contexte de TON site. Donc avec acces a :
  - cookies de session (vol de session)
  - localStorage / IndexedDB (tokens, données)
  - actions au nom de la victime (changer mot de passe, virer argent)

  **3 sous-types :**

  1. **Stored XSS (le pire)** : le payload est stocke en BDD (commentaire,
     post forum, message chat) et execute pour TOUS les visiteurs.
     Cas reel : MySpace 2005 (Samy worm, 1M+ comptes en 20h).

  2. **Reflected XSS** : le payload est dans l'URL (querystring), execute
     uniquement pour les visiteurs qui cliquent sur le lien malicieux.
     Vecteur : phishing avec lien arrange.

  3. **DOM-based XSS** : le payload exploite du JS côté client qui
     manipule le DOM avec des inputs externes (location.hash,
     postMessage, etc.). Pas vu par le serveur.

  **Defenses (par ordre de priorité) :**

  1. **Output encoding** : echapper les caracteres spéciaux a l'AFFICHAGE,
     selon le contexte.
     - HTML body : `<` `>` `&` `"` `'`
     - HTML attribute : encoder differemment selon quoted/unquoted
     - JavaScript : `\xHH` encoding
     - Frameworks modernes : React/Vue/Svelte le font AUTO sauf dans
       `dangerouslySetInnerHTML` / `v-html` / `{@html}`. Evite ces
       echappatoires.

  2. **CSP (Content Security Policy)** : header HTTP qui restreint les
     scripts autorises. Defense-in-depth si l'output encoding rate.
     ```
     Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'
     ```
     - **`nonce-*`** : seul les scripts avec ce nonce s'executent
     - **strict-dynamic** : scripts dynamiquement crees par d'autres
       scripts trustes

  3. **HttpOnly cookies** : cookie de session inaccessible au JS, donc
     pas vol par XSS (`document.cookie` ne le voit pas). **OBLIGATOIRE**.

  4. **Sanitization** (en dernière ligne) : si tu DOIS afficher du HTML
     user (commentaires riches, markdown), utilise `DOMPurify` ou une
     allowlist stricte de tags.

  **CSRF - Cross-Site Request Forgery**

  L'attaquant cree une page sur evil.com qui declenche une requete vers
  yourapp.com en utilisant les cookies de session de la victime
  (envoyes automatiquement par le browser car même origine).

  La victime est connectee a yourapp.com, visite evil.com, son browser
  envoie la requete avec les cookies, le serveur croit que c'est une
  action legitime.

  **Defenses (par ordre de priorité) :**

  1. **Cookie SameSite (le plus important en 2026)** :
     ```
     Set-Cookie: session=...; SameSite=Lax; Secure; HttpOnly
     ```
     - **`Lax`** : cookies non envoyes sur les POST cross-origin (defense
       suffisante pour 95% des cas). Default depuis Chrome 80 (2020).
     - **`Strict`** : cookies non envoyes sur AUCUNE navigation
       cross-origin (peut casser les liens entrants).

  2. **CSRF tokens** : token random dans le formulaire, valide par le
     serveur. Synchronizer pattern.
     - Stocke en session OU en double-submit cookie.
     - Toujours utile pour les apps qui ne peuvent pas reposer sur
       SameSite (legacy, embed iframes, navigateurs anciens).

  3. **Header `Origin` / `Referer`** : le serveur verifie que le POST
     vient bien du domaine attendu. Fonctionne sauf si l'attaquant fait
     un cross-site POST avec same-origin (rare).

  4. **POST only pour les actions sensibles** : ne JAMAIS faire `GET
     /api/users/promote?id=42` - les GET sont triggables par n'importe
     quel `<img>` ou `<a>`.

  **La trinite cookies 2026 :**

  ```
  Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax; Path=/
  ```

  - **HttpOnly** : empeche XSS de voler le cookie
  - **Secure** : envoye uniquement en HTTPS
  - **SameSite=Lax** : empeche CSRF cross-origin

  Cette ligne, par defaut, dans tous tes apps. NextAuth/Express-session/
  Django/Rails l'ont déjà par defaut depuis ~2022. Verifie en prod.

  **Test en code review :**

  - Un `dangerouslySetInnerHTML` (React), `v-html` (Vue), `{@html}`
    (Svelte) ? **Stop, demande pourquoi c'est necessaire**.
  - Un cookie sans `SameSite` ? **Stop, ajoute Lax au minimum**.
  - Une route POST qui change un état important sans CSRF token et
    sans `SameSite=Strict` ? **Stop, ajoute la defense**.
