debrief: |
  La **vérité brutale** des sauvegardes : **30% des sauvegardes
  echouent au test de restauration** (etude Veeam 2023). Si tu
  ne testes pas, tu ne sais pas si ta sauvegarde marche.

  Le drame du jour J : "on a des sauvegardes" -> "ah les
  sauvegardes sont corrompues / incomplete / ne fonctionnent
  pas".

  **Pourquoi les sauvegardes echouent souvent :**

  **Cause 1 : Corruption silencieuse**

  - Erreurs disque (bit rot)
  - Erreurs de transfert non detectees
  - Backup software bug
  - Detectee uniquement au moment de restaurer

  **Cause 2 : Espace insuffisant**

  - La sauvegarde a echoue depuis X mois car disque plein
  - Personne n'a verifie les alertes
  - Sauvegarde incomplete

  **Cause 3 : Mots de passe / credentials perdus**

  - Mot de passe de l'archive chiffree perdu
  - Compte cloud mort (employé parti)
  - Cle de chiffrement perdue

  **Cause 4 : Software incompatible**

  - Sauvegarde faite sur Windows Server 2012
  - Restauration tentee sur Windows Server 2022
  - Format incompatible

  **Cause 5 : Procédure non-documentee**

  - Personne ne sait comment restaurer
  - Le sysadmin qui savait est parti
  - Aucune documentation

  **Cause 6 : Sauvegarde mais pas tout**

  - On sauvegardait C:\données mais pas D:\fichiers_clients
  - Configurations / paramètres pas sauvegardes
  - Bases de données mal exportees

  **Cause 7 : Sauvegarde compromise**

  - Ransomware a chiffre les sauvegardes (cas 80% des
    ransomware en 2023)
  - Sauvegarde NAS connectee au réseau

  **Tous ces cas** : detecte uniquement par test de
  restauration. Aucun moyen de savoir avant.

  **Le programme de tests recommande :**

  **Test 1 : Restauration d'un fichier - MENSUEL**

  Objectif : valider que les sauvegardes fonctionnent au niveau
  fichier.

  Procédure :
  1. Choisir un fichier aleatoire dans la sauvegarde
  2. Restaurer dans un dossier temporaire
  3. Vérifier l'integrite (ouvrir, comparer)
  4. Documenter

  **Duree** : 15-30 minutes
  **Outils** : interface du logiciel de sauvegarde

  **Test 2 : Restauration d'un serveur entier - TRIMESTRIEL**

  Objectif : valider que la sauvegarde complete d'un serveur
  fonctionne (boot, services, données).

  Procédure :
  1. Restaurer sur un serveur de test (VM dediee)
  2. Booter
  3. Vérifier les services (BDD, web, fichiers)
  4. Tester les acces utilisateurs
  5. Documenter

  **Duree** : 2-4 heures
  **Outils** : logiciel sauvegarde + Hyper-V / VMware test

  **Test 3 : Simulation d'incident complet - ANNUEL**

  Objectif : valider la procédure de reprise d'activite (PRA).

  Procédure :
  1. Scenario : ransomware total (toutes les machines
    chiffrees)
  2. Restauration depuis off-site (cloud)
  3. Tous les services retablis
  4. Tous les utilisateurs peuvent travailler
  5. Mesurer le RTO (temps de retour a la normale)
  6. Mesurer le RPO (perte de données)

  **Duree** : 1-3 jours (simulation, pas en prod)
  **Implication** : équipe IT entiere + management

  **Format** : weekend / soir pour ne pas perturber

  **Documentation des tests :**

  Pour chaque test :

  | Élément | Description |
  |---|---|
  | Date | YYYY-MM-DD |
  | Type de test | Fichier / Serveur / Complet |
  | Cible | Quel serveur / quelle BDD |
  | Volume | Taille des données |
  | Duree | Temps de restauration |
  | Résultat | Succes / Echec |
  | Anomalies | Description |
  | Actions correctives | Si echec |

  **Conservation** : 3 ans minimum (audit NIS2).

  **Indicateurs cles (KPI) :**

  - **Taux de succes des tests** : doit être 100% (sinon
    investigation)
  - **RTO mesure** : durée de restauration (doit être dans la
    cible)
  - **RPO mesure** : perte de données (doit être dans la cible)
  - **Frequence des tests** : conformité au calendrier

  **Outils pour automatiser les tests :**

  **a) Veeam SureBackup** (commercial)
  - Test automatique mensuel
  - Boot des VMs sauvegardees en sandbox
  - Vérification des services
  - Rapport automatique

  **b) Microsoft DPM / Azure Backup** (Microsoft)
  - Tests de coherence automatiques
  - Vérification mensuelle

  **c) Scripts maison** (gratuit)
  - PowerShell / Bash pour tests basiques
  - Restauration auto d'un fichier test
  - Comparaison de hash

  **d) Test "Chaos Engineering"**
  - Netflix Chaos Monkey
  - Tests volontaires de pannes en prod
  - Pour les organisations matures

  **Le piege "ça marche ailleurs, donc ça marche ici" :**

  Chaque environnement est unique. Une sauvegarde qui marche
  chez Acme Corp peut echouer chez Beta SAS pour des raisons
  spécifiques (firewall, credentials, version logiciels).

  Solution : **tester DANS TON ENVIRONNEMENT**.

  **Le piege "on a une assurance cyber, ça couvre" :**

  L'assurance cyber peut **refuser de payer** si :
  - Tu ne peux pas demontrer une procédure de sauvegarde
    documentee
  - Tu ne peux pas demontrer des tests reguliers
  - Tu n'as pas conserve les preuves

  Solution : test + documentation = condition d'application
  de l'assurance.

  **Le piege "on teste si on a le temps" :**

  Le temps des tests **est** le temps de la cyber. Si tu n'as
  pas le temps de tester :
  - Tu n'as pas le temps de la cyber
  - L'incident sera 1000x plus chronophage
  - Sans tests, tu **ne sais pas** si tu peux restaurer

  Allocation : 4-8 heures / mois pour les tests.

  **Procédure d'urgence si test echoue :**

  Si un test revele un echec :

  **Étape 1 : Investigation immediate**
  - Pourquoi ça a echoue ?
  - Depuis quand ?
  - Quelles données affectees ?

  **Étape 2 : Remediation rapide**
  - Restaurer depuis une autre sauvegarde (multi-versions)
  - Reconfigurer le système de sauvegarde
  - Lancer une nouvelle sauvegarde

  **Étape 3 : Documentation**
  - Cause racine
  - Actions correctives
  - Plan pour éviter recidive

  **Étape 4 : Communication**
  - Information au RSSI / DG
  - Si données personnelles potentiellement perdues : peut
    être violation RGPD a notifier

  **Cas reels :**

  **Cas 1 - Maersk 2017 (NotPetya, $300M)**

  Geant maritime victime du ransomware NotPetya. Tous les
  systèmes chiffres. Heureusement, **un seul** serveur sauvegarde
  d'Active Directory était offline a cause d'une coupure de
  courant (Ghana). Restauration depuis ce serveur unique.

  Diagnostic : avaient des sauvegardes mais aucun test
  systematique. Ce serveur miracle a sauve l'entreprise.

  Cout total : **300 M USD** + 10 jours d'arrêt.

  Lecon : tester régulièrement, pas compter sur la chance.

  **Cas 2 - PME française 2022 (test réussi)**

  PME 100 personnes victime ransomware. Sauvegardes :
  - NAS local : chiffre par ransomware
  - Cloud Backblaze : intact

  Procédure de restauration documentee + tests trimestriels
  -> restauration en 18h sans problème.

  **Pas de paiement** rançon. Cout : 5K EUR (perte d'activite).

  ROI des tests : évident.

  **Cas 3 - Hôpital US 2018 (sauvegardes corrompues)**

  Hôpital pense avoir des sauvegardes pendant 6 mois. Ransomware
  : besoin de restaurer. Decouverte : sauvegardes corrompues
  depuis 5 mois (espace insuffisant, sauvegardes incompletes).

  Résultat : paiement de 600 K USD de rançon + procédures
  legales + reputation degradee.

  Lecon : test mensuel aurait detecte 5 mois plus tot.

  **Le calendrier modèle pour PME :**

  | Élément | Frequence | Responsable | Duree |
  |---|---|---|---|
  | Restauration fichier | Mensuelle (1er lundi) | Admin IT | 30 min |
  | Restauration serveur | Trimestrielle | Admin IT | 2-4h |
  | Simulation incident | Annuelle (octobre) | Équipe + RSSI | 1-3 jours |
  | Audit sauvegardes | Annuel | RSSI + auditeur externe | 5 jours |

  **Documentation requise** : journal de tests, certificats,
  plans d'action.

  **La règle d'or :**

  > **Une sauvegarde non-testee N'EST PAS une sauvegarde. 30%
  > echouent au test (Veeam 2023). Tests progressifs : fichier
  > mensuel, serveur trimestriel, complet annuel. Documentation
  > obligatoire. Sans tests = Maersk 300M$. Avec tests = ROI
  > évident.**
