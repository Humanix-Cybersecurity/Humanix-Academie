debrief: |
  Le **RTO** et le **RPO** sont les 2 chiffres fondateurs de
  la stratégie de continuite d'activite. Sans eux, on ne sait
  pas combien dimensionner.

  **Definitions :**

  **RTO** (Recovery Time Objective)
  = **durée maximum d'indisponibilite toleree** après un
  incident.

  Exemples :
  - RTO = 1 heure : le service doit être back en 1h max
  - RTO = 24 heures : tolerable d'être down 24h
  - RTO = 1 semaine : pas critique

  **RPO** (Recovery Point Objective)
  = **quantité maximum de données perdues** acceptable.

  Exemples :
  - RPO = 0 : aucune perte de données toleree (replication
    temps reel)
  - RPO = 1 heure : on tolere de perdre les modifications
    de la dernière heure
  - RPO = 24 heures : on accepte de perdre 1 jour de données

  **L'arbitrage RTO/RPO vs coût :**

  Plus les RTO/RPO sont courts, plus c'est cher :

  | RTO | RPO | Solution typique | Cout |
  |---|---|---|---|
  | **0 / 0** | Replication temps-reel + standby actif | x10 |
  | **1h / 15min** | Replication async + sauvegarde fréquente | x5 |
  | **4h / 1h** | Sauvegarde horaire + restoration rapide | x3 |
  | **24h / 4h** | Sauvegarde quotidienne | x1 (baseline) |
  | **72h / 24h** | Sauvegarde hebdomadaire | x0.7 |
  | **1 semaine / 7 jours** | Sauvegarde hebdo, archive | x0.5 |

  Le defi : **trouver le bon compromis** entre besoin business
  et coût.

  **Methodologie de calcul :**

  **Étape 1 : Cartographier les services**

  Lister tous les services / applications de l'entreprise :
  - ERP / CRM
  - Email
  - Site web public
  - Téléphone (VoIP)
  - Fichiers partages
  - BDD clients
  - Outils de production
  - Etc.

  **Étape 2 : Evaluer la criticite business de chaque service**

  Questions a poser :
  - Si ce service est down 1h, impact business ?
  - Si on perd les données de la dernière heure, impact ?
  - Combien d'utilisateurs sont impactes ?
  - Combien on perd par heure d'indisponibilite ?

  **Classification standard** :

  | Tier | Description | RTO typique | RPO typique |
  |---|---|---|---|
  | **Tier 0 - Vital** | Sans = entreprise s'arrete | < 1h | < 15 min |
  | **Tier 1 - Critique** | Important pour business | 1-4h | 1h |
  | **Tier 2 - Important** | Pas vital mais sensible | 4-24h | 4-24h |
  | **Tier 3 - Standard** | Peut attendre | 24-72h | 24h |
  | **Tier 4 - Faible** | Archives, peu utilise | 1 semaine | 1 semaine |

  **Étape 3 : Calculer le coût d'indisponibilite**

  Cout d'1 heure d'arrêt :

  - Salaires perdus : (Nb employés affectes) × (Salaire moyen)
    / horaire
  - Chiffre d'affaires perdu : CA mensuel / 730h × heures
    affectees
  - Penalites contractuelles : SLA clients
  - Cout de remediation : intervention urgence
  - Cout reputation : perte de clients

  **Exemple PME 80 personnes (CA 12M) :**
  - Salaires : 80 × 25 EUR/h = 2000 EUR/h (si tout le monde
    bloque)
  - CA : 12M / 730 = 16K EUR / mois = 16 EUR / heure (mais
    perdu si 24h+)
  - **Cout 1h ERP down** : ~3-5 K EUR

  Si l'ERP doit être down 24h : 75-120 K EUR de perte.

  **Étape 4 : Comparer aux RTO/RPO actuels**

  Pour chaque service, compare :
  - RTO **actuel** (mesure realiste avec sauvegardes
    actuelles)
  - RTO **souhaite** (selon analyse business)
  - **Ecart** : RTO actuel > RTO souhaite ? Si oui,
    investissement necessaire.

  **Étape 5 : Plan d'action**

  Selon les ecarts identifies, dimensionner les sauvegardes
  et la continuite :

  - Si **ecart faible** : peut-être juste optimisation
    (procédure plus rapide, automation)
  - Si **ecart important** : investissement plus consequent
    (replication, standby)

  **Exemple complet pour PME 80 personnes :**

  | Service | RTO souhaite | RPO souhaite | Solution actuelle | Ecart | Action |
  |---|---|---|---|---|---|
  | **ERP comptable** | 4h | 1h | Sauvegarde quotidienne (RTO 24h, RPO 24h) | Important | Sauvegarde horaire + replication async |
  | **Email** | 24h | 4h | Microsoft 365 (RTO 1h, RPO 1h) | OK | Maintien actuel |
  | **CRM** | 8h | 4h | Salesforce (RTO 4h, RPO 1h selon plan) | OK | Maintien actuel |
  | **Fichiers partages** | 24h | 4h | OneDrive Business (RTO 4h, RPO 1h) | OK | Maintien actuel |
  | **Site web public** | 1h | 0 | Hebergeur (RTO 1h, RPO 1h) | RPO ecart | Replication BDD |
  | **Voix VoIP** | 8h | N/A | Operateur (RTO 4h) | OK | Maintien actuel |
  | **Archives** | 1 semaine | 1 semaine | Sauvegarde hebdomadaire | OK | Maintien actuel |

  **Plan d'action** : focus ERP comptable (le plus critique +
  le plus en ecart) + replication BDD site web.

  **Cout estime** :
  - Sauvegarde horaire ERP : +50 EUR/mois (cloud)
  - Replication async ERP : +200 EUR/mois (Veeam ou similaire)
  - Replication BDD site : +30 EUR/mois
  - **Total** : +280 EUR/mois pour atteindre les cibles

  **Comparaison vs coût d'indisponibilite** :
  - Cout 24h ERP down (sans amelioration) : 100K EUR (1
    incident/an = 100K EUR)
  - Cout amelioration : 3.4K EUR/an
  - **ROI** : 30x

  **Le piege "RTO 0 c'est toujours mieux" :**

  FAUX. RTO 0 implique :
  - **Replication synchrone** : x5 le coût des serveurs
  - **Standby actif** : x2 les licences
  - **Opérations** complexes (synchronisation, basculement)
  - **Maintenance** plus difficile

  Justifie uniquement pour services TIER 0 (banque, sante,
  defense). Pour PME standard, RTO 4h-24h est acceptable.

  **Le piege "tous les services ont le même RTO/RPO" :**

  FAUX. Différents services ont différentes criticites :
  - L'ERP comptable : critique (RTO court)
  - Les archives : non critique (RTO long)

  Avoir un RTO unique = surdimensionnement (cher) ou
  sous-dimensionnement (risque).

  **Le piege "le RPO se mesure en jours" :**

  Pour données volatiles (BDD transactionnelle, ERP), RPO en
  jours = catastrophe (perte d'1 jour de transactions = perte
  de crédibilité + clients).

  Pour archives statiques, RPO en jours acceptable.

  **Solutions techniques selon RTO/RPO :**

  **RTO 0 / RPO 0 (continuite parfaite)**
  - **Replication synchrone** entre 2 datacenters
  - **Standby actif** : serveur en miroir, prend le relais en
    cas de panne
  - **Cluster active-active** : repartit la charge
  - **Cout** : x10 cost baseline

  **RTO 1h / RPO 15 min (haute disponibilite)**
  - **Replication asynchrone** : copies fréquentes
  - **Standby passif** : serveur eteint, demarre en cas de
    panne
  - **Failover automatique**
  - **Cout** : x5 baseline

  **RTO 4h / RPO 1h (PME critique)**
  - **Sauvegarde horaire** + restoration rapide
  - **Hot spare** : matériel pret a déployer
  - **Procédure documentee + testee**
  - **Cout** : x3 baseline

  **RTO 24h / RPO 4h (PME standard)**
  - **Sauvegarde quotidienne** ou plusieurs fois / jour
  - **Restoration en quelques heures**
  - **Procédure documentee**
  - **Cout** : x1 baseline (le standard)

  **RTO 72h / RPO 24h (PME très petite, archives)**
  - **Sauvegarde quotidienne** standard
  - **Restoration manuelle**
  - **Cout** : x0.7 baseline

  **Les outils de mesure :**

  - **Zerto** : replication continue + RTO/RPO temps reel
  - **Veeam Backup Replication** : standard PME
  - **AWS / Azure** : services natifs (Site Recovery,
    Backup)
  - **VMware vSphere Replication** : pour clusters

  **Cas reels :**

  **Cas 1 - PME 2023 (analyse RTO/RPO bien faite)**

  PME 100 employés a fait l'analyse complete RTO/RPO.
  Decouverte : ERP cible 4h mais reel 48h. Investissement :
  Veeam Replication = 5K EUR/an.

  Résultat : ransomware 6 mois plus tard, ERP restaure en 3h.
  ROI : évident (vs 100K EUR de perte si 48h down).

  **Cas 2 - Hôpital US 2022 (pas d'analyse)**

  Hôpital n'avait pas d'objectif RTO documente. Ransomware =
  72h de degradation (parts critiques) + 2 semaines de
  rétablissement total.

  Cout : 1.5 M USD + reputation + procédures.

  Avec analyse + investissement (200K USD initial), aurait
  été RTO 8h max + RPO 1h.

  **Cas 3 - Banque française 2024 (analyse rigoureuse)**

  Banque avec analyse RTO/RPO précis :
  - Tier 0 (paiements) : RTO 30 min, RPO 0 (replication
    synchrone)
  - Tier 1 (consultations) : RTO 4h, RPO 15 min
  - Tier 2 (BO, RH) : RTO 24h, RPO 4h
  - Tier 3 (archives) : RTO 1 semaine, RPO 24h

  Investissement : 5M EUR sur 3 ans. ROI prouve par tests
  reguliers.

  **NIS2 et RTO/RPO :**

  L'article 21 mesure 3 (continuite des activites) impose :
  - **Définition documentee** des RTO et RPO
  - **Plan de continuite** (PCA)
  - **Plan de reprise** (PRA)
  - **Tests** reguliers (cf ep 02)

  Pour les entites essentielles : audit ANSSI verifie ces
  points.

  **Outils gratuits pour le calcul :**

  - **Excel / Google Sheets** : tableau RTO/RPO simple
  - **CISO Assistant** (intuitem, OSS) : module de continuite
    avec RTO/RPO
  - **EBIOS RM** (ANSSI) : analyse de risques avec impact

  **La règle d'or :**

  > **RTO et RPO sont spécifiques a chaque service. Analyse
  > business préalable obligatoire (criticite, coût
  > indisponibilite). Solution dimensionnee en conséquence
  > (éviter sur ou sous-dimensionnement). Documentation NIS2.
  > Tests reguliers (cf ep 02). RTO 0 = uniquement pour
  > services vitaux (coûts x10).**
