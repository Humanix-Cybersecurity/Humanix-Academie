debrief: |
  L'authentification email est une defense **critique** mais souvent
  ignoree. 3 protocoles complémentaires : SPF, DKIM, DMARC.

  **Le problème : SMTP autorise tout le monde a se faire passer pour
  qui il veut**

  Le protocole SMTP (1982) ne fait AUCUNE vérification d'identité. Tu
  peux envoyer un email avec `From: pdg@apple.com` depuis n'importe
  quel serveur si le destinataire ne verifie rien. C'est l'origine de
  90% du phishing.

  **Couche 1 - SPF (Sender Policy Framework, 2006)**

  Tu publis dans ton DNS la liste des IPs/serveurs autorises a envoyer
  pour ton domaine. Le serveur destinataire verifie que l'email vient
  bien d'une de ces IPs.

  ```
  humanix-academie.fr. IN TXT "v=spf1 include:_spf.google.com ~all"
  ```

  - `include:_spf.google.com` : autorise les serveurs Google Workspace.
  - `~all` : tout le reste = soft fail (note suspect, mais delivre).
  - `-all` : tout le reste = hard fail (rejette).

  **Limites SPF** :
  - Casse en cas de forwarding (forward email = email envoye depuis
    le serveur du forwarder, plus le bon SPF).
  - Verifie le `MAIL FROM` SMTP (envelope), pas le `From` visible
    dans le client - donc spoof du `From` reste possible.

  **Couche 2 - DKIM (DomainKeys Identified Mail, 2007)**

  Le serveur emetteur signe cryptographiquement l'email avec sa cle
  privée. Le destinataire verifie la signature avec la cle publique
  publiee dans le DNS.

  ```
  default._domainkey.humanix-academie.fr. IN TXT "v=DKIM1; k=rsa; p=MIGfMA0..."
  ```

  L'email contient un header :
  ```
  DKIM-Signature: v=1; a=rsa-sha256; d=humanix-academie.fr;
                  s=default; h=From:Subject:Date; b=ABC123...
  ```

  Avantages DKIM :
  - Survit au forwarding (la signature reste valide).
  - Garantit integrite (modification du contenu casse la signature).
  - Verifie le `From` visible (header).

  **Couche 3 - DMARC (Domain-based Message Authentication, 2012)**

  C'est le chef d'orchestre. DMARC dit : "tu dois passer SPF OU DKIM
  pour le domaine `From`, sinon voici ce qu'on fait".

  ```
  _dmarc.humanix-academie.fr. IN TXT "v=DMARC1; p=reject;
                                       rua=mailto:dmarc@humanix.fr;
                                       ruf=mailto:forensic@humanix.fr"
  ```

  Les 3 politiques `p=` :

  - **`p=none`** : monitoring uniquement. Recoit les rapports
    d'agregation (`rua=`) sur qui envoie quoi avec ton domaine. Aucun
    blocage. **A utiliser SEULEMENT en phase de mise en place pour
    decouvrir les emetteurs legitimes.**

  - **`p=quarantine`** : les emails qui echouent SPF+DKIM vont en
    spam. Defense raisonnable, ne casse pas trop si un emetteur legit
    est oublie.

  - **`p=reject`** : les emails qui echouent SPF+DKIM sont REJETES
    par le serveur destinataire (jamais delivres). **Defense maximale.**

  **Le rapport DMARC (rua=)**

  Chaque jour, les serveurs destinataires (Gmail, Microsoft, Yahoo)
  t'envoient un fichier XML avec les statistiques :
  - Nb d'emails reçus avec ton domaine en From
  - Combien ont passe SPF / DKIM
  - IPs des emetteurs (legitimes ou attaquants)

  Outils pour les lire facilement :
  - **dmarcian** (fr.dmarcian.com) : analyse + dashboard
  - **Postmark DMARC Digest** (gratuit)
  - **Valimail Monitor** (gratuit)

  **Migration DMARC en 4 étapes (procédure recommandee) :**

  1. **Étape 0 (J)** : publier SPF (`~all`) et DKIM. Tester avec
     `mail-tester.com` ou `mxtoolbox.com`.

  2. **Étape 1 (J+0)** : publier `p=none` avec `rua=`. Recevoir les
     rapports pendant 2-4 semaines pour identifier tous les emetteurs
     legitimes (CRM, marketing, ERP, employés individuels via Outlook
     externe...).

  3. **Étape 2 (J+30)** : passer en `p=quarantine` avec `pct=10`
     (10% du trafic est mis en quarantaine, le reste est en p=none).
     Augmenter progressivement (`pct=25`, `pct=50`, `pct=100`) sur
     2-4 semaines.

  4. **Étape 3 (J+90)** : passer en `p=reject; pct=100`. Defense
     maximale. Les emails frauduleux sont bloques avant arrivée.

  **Outils de diagnostic gratuits :**

  - **mxtoolbox.com/dmarc** : verifie ta config
  - **mail-tester.com** : envoie un email a une adresse, recois un
    score complet (SPF, DKIM, DMARC, contenu, headers).
  - **dmarcian** : analyse des rapports DMARC
  - **dig** en CLI :
    ```
    dig +short TXT _dmarc.humanix-academie.fr
    dig +short TXT humanix-academie.fr | grep spf1
    ```

  **Cas reel (anonymise) :**

  PME française du BTP (50 employés), client Sage. Un attaquant
  spoofe `directeur@sage-clientfr.example` (vrai domaine du client),
  envoie un email "le RIB du fournisseur a change" a la compta. Vir
  de 87 000 EUR. La compta verifie le From - tout semble legitime.

  Investigation : SPF du client = inexistant. DMARC = inexistant.
  N'importe quel SMTP au monde pouvait envoyer comme `directeur@`. Le
  client a paye - juridiquement, c'est sa faute (il n'a pas mis en
  place les mesures techniques attendues en 2024).

  **Pour ton domaine, AUJOURD'HUI :**

  1. Verifie : `dig TXT _dmarc.tondomaine.fr`. Si vide ou `p=none`
     depuis plus de 30 jours = problème.
  2. Si tu utilises Gmail/Microsoft 365 : leur admin documente comment
     publier SPF/DKIM/DMARC. ~10 minutes de setup.
  3. Active les rapports DMARC (`rua=`) vers une adresse dediee.
