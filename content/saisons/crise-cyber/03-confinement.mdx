debrief: |
  Le **confinement** est l'etape la plus critique d'une reponse a
  incident. Mal fait = degats massifs ou perte definitive des preuves
  pour comprendre / prouver / poursuivre.

  **Les 3 objectifs simultanes du confinement :**

  1. **STOP l'attaque en cours** : couper l'exfiltration, le lateral
     movement, le ransomware encore en chiffrement.
  2. **PRESERVE les preuves** : ne pas detruire la RAM, les logs, les
     fichiers temporaires - tout ce qui permet de reconstituer
     l'attaque.
  3. **NE PAS aggraver** : eviter de declencher des comportements
     defensifs du malware (auto-destruction, chiffrement final).

  **Les 4 erreurs classiques (a NE PAS faire) :**

  ❌ **Redemarrer la machine**
  - Detruit la RAM (preuves)
  - Peut declencher chiffrement final ransomware
  - Peut effacer les logs en cours d'ecriture

  ❌ **Eteindre brutalement**
  - Detruit la RAM (preuves)
  - Peut corrompre les filesystems (perte de donnees secondaires)
  - Reaction stress, pas reflexion

  ❌ **Formater / re-imager**
  - Detruit toutes les preuves
  - Pas d'investigation possible
  - Tu ne sais pas si le malware persiste ailleurs

  ❌ **Continuer a utiliser la machine "doucement"**
  - L'attaquant continue son boulot
  - Le ransomware finit son chiffrement
  - L'exfiltration continue

  **La procedure correcte (par type d'incident) :**

  **Cas 1 - Compromission de poste utilisateur (PC CFO)**

  1. **Deconnecter du reseau** (cable Ethernet debranche + WiFi
     desactive + Bluetooth off). NE PAS eteindre.
  2. **Si possible : RAM dump** (FTK Imager, DumpIt) sur cle USB
     externe. Capture l'image memoire.
  3. **Disk image** : creer une image bit-a-bit du disque (dd,
     ddrescue, FTK Imager) pour analyse forensic.
  4. **Reset des credentials** depuis un AUTRE poste : mot de passe,
     MFA, cookies de session OAuth (revoke tokens M365/Google).
  5. **Mettre la machine en quarantaine** physique (etiquette "ne pas
     utiliser - investigation en cours").

  **Cas 2 - Compromission de serveur (ERP Linux)**

  1. **Snapshot de la VM** (si virtualisee : VMware, Proxmox, AWS).
     C'est l'equivalent du dump RAM + disk pour les VMs.
  2. **Isoler du reseau** : firewall rule pour bloquer toutes les
     connexions sortantes ET entrantes (sauf admin pour
     investigation).
  3. **Auditer les comptes nouveaux** : `lastlog`, `/etc/passwd`,
     `/etc/sudoers.d/`, `~/.ssh/authorized_keys` de tous les users.
  4. **Verifier les logs** AVANT remediation : `journalctl`,
     `/var/log/auth.log`, `~/.bash_history` (peut etre delete par
     l'attaquant).
  5. **NE PAS DELETE les comptes malicieux** tant que l'investigation
     n'est pas faite. Juste les disable (`usermod -L`).

  **Cas 3 - Ransomware en cours de chiffrement (serveur backup)**

  C'est le seul cas ou eteindre IMMEDIATEMENT peut etre justifie :
  - Si l'attaquant est en train de chiffrer activement, chaque
    seconde de plus = plus de fichiers perdus
  - Eteindre brutallement (hard shutdown) = perdre la RAM mais sauver
    les fichiers non encore chiffres
  - Ensuite : analyse forensic depuis l'image disque (RAM perdue
    mais c'etait le moindre mal)

  Sequence : 1) deconnecter du reseau, 2) si chiffrement actif a
  haute vitesse : eteindre brutallement, 3) image du disque, 4)
  investigation.

  **Cas 4 - Webshell sur serveur web (Mac compta)**

  1. **Isoler du reseau** (deconnecter Ethernet + WiFi)
  2. **NE PAS DELETE le webshell** tant que pas analyse - capture sa
     fonctionnalite (quels endpoints, quelles capacites).
  3. **Audit complet** des fichiers crees/modifies recemment (find
     -mtime -7).
  4. **Reset les credentials** de tout compte qui aurait pu etre
     vole via le webshell.
  5. **Re-image** le serveur a partir d'une sauvegarde verifiee
     saine.

  **Outils forensic gratuits :**

  - **FTK Imager** : capture RAM + disk, gratuit
  - **DumpIt** : RAM dump Windows minimaliste, gratuit
  - **Volatility 3** : analyse RAM (open source)
  - **GRR Rapid Response** : forensic distribue (Google OSS)
  - **Velociraptor** : monitoring + forensic (open source)
  - **Wireshark** : analyse trafic reseau

  **Le piege "j'ai un EDR, je m'en remets a lui" :**

  Les EDR (Microsoft Defender, CrowdStrike, SentinelOne, HarfangLab)
  peuvent isoler une machine via leur console. Pratique mais :
  - Verifie que l'isolation a bien fonctionne (test ping)
  - Si l'EDR est lui-meme compromis (rare mais possible), il peut
    cacher l'incident
  - Combiner EDR isolation + deconnexion reseau physique pour les
    cas les plus critiques

  **Le timeline forensic :**

  Pendant le confinement, NOTER :
  - Heure exacte de detection
  - Heure exacte de chaque action (deconnexion, dump, etc.)
  - Quelle action sur quelle machine
  - Etat avant/apres

  Ce timeline servira a :
  - Recreer la chronologie (forensic)
  - Justifier les actions devant CNIL/avocats
  - REX pour la prochaine fois

  **Note legale : contacter les autorites**

  Pour les attaques significatives, signaler a :
  - **Gendarmerie cyber (C3N)** : pour plainte penale
  - **ANSSI / CERT-FR** : pour conseil et alerte de la communaute
  - **CNIL** : si donnees personnelles affectees (cf. ep 04)

  Ces signalements n'empechent pas l'investigation interne, mais
  peuvent apporter expertise + traçabilite.
