debrief: |
  L'**offboarding des prestataires des outils de collaboration**
  est une étape cruciale souvent negligee.

  Verizon DBIR : 60% des breaches insider partent d'un compte
  non-revoque correctement. Les outils de collaboration sont
  parmi les plus oublies (a côté de VPN, AD, SaaS).

  **Pourquoi c'est critique :**

  Un prestataire avec acces chat residuel peut :

  - **Voir** les conversations en cours
  - **Telecharger** fichiers historiques
  - **Être exploite** par un attaquant qui compromet son
    compte
  - **Fournir info** au concurrent qui le recrute (cf saison
    cyber-rh ep 04)

  Ce risque est **constant** tant que l'acces est actif.

  **La checklist exhaustive offboarding chat :**

  **Outils de chat / collaboration a couper :**

  **1. Slack**
  - Liste canaux ou Bob est membre
  - **Workspace** : remove from workspace
  - **Sortie** : tous les channels
  - **DMs** : peuvent rester (mais l'autre côté n'a plus
    contact)
  - **Audit log** : dernière connexion

  **2. Microsoft Teams**
  - Liste équipes / canaux
  - **Guest Access** : remove
  - **External access** (federation) : remove
  - **Audit log** Teams

  **3. Notion**
  - Liste pages / spaces partages
  - **Member / Guest** status : remove
  - **Permissions** : revoke explicit

  **4. SharePoint / OneDrive**
  - Liste sites / dossiers partages
  - **Permissions** : revoke
  - **Liens partages** explicitement avec lui : revoke

  **5. Confluence / Jira**
  - User account : disable
  - **Anonymous access** sur ses pages

  **6. Discord / Mattermost**
  - Server membership : remove
  - **Channels** : remove

  **7. Google Workspace** (si utilise)
  - **Google Drive** : remove from shared
  - **Google Meet** : remove from organizational guests
  - **Google Chat** : remove

  **8. GitHub / GitLab / Bitbucket**
  - **Organization membership** : remove
  - **Repository access** : remove
  - **Personal access tokens** : revoke
  - **SSH keys** : remove

  **9. Outils projets**
  - Asana, Trello, Monday, ClickUp, Linear : remove
  - **Apps tierces** Slack/Teams qu'il a installees
  - **Integrations** Zapier qu'il a configurees

  **10. Outils dev**
  - Sentry, Datadog, New Relic
  - Stripe, AWS, Azure (si dev)

  **11. CRM / Sales**
  - Hubspot, Salesforce, Pipedrive

  **12. Outils marketing**
  - Mailchimp, Brevo, Sendinblue

  **Procédure d'offboarding chat :**

  **Étape 1 : Pre-departure (J-7)**

  - **Inventaire** des acces du prestataire
  - **Coordination** avec son manager
  - **Liste de canaux/équipes** ou il était

  **Étape 2 : Jour J (depart effectif)**

  Action **dans l'heure** suivant le depart :

  - **Slack** : remove from workspace (workspace settings)
  - **Teams** : remove guest from all teams
  - **Notion** : remove from workspace
  - **SharePoint** : remove from sites
  - **GitHub** : remove from organization
  - **Tous autres outils** identifies

  **Étape 3 : J+1 a J+7 (audit proche)**

  - **Vérification** : aucune connexion residuelle
  - **Logs Slack / Teams** : analyse
  - **Comptes fantomes** : détection

  **Étape 4 : J+30 (audit final)**

  - **Audit complet** : aucun acces actif sur aucun outil
  - **Documentation** finale
  - **Lessons learned**

  **Cas particuliers :**

  **Cas 1 : Slack DMs**

  Après revocation Slack :
  - **DMs** historiques restent visibles côté
    employé interne (l'autre côté)
  - **Bob** ne peut plus envoyer / recevoir
  - **Information sensible** dans DMs : reste visible

  Action : sensibilisation équipe (pas de mots de passe / info
  sensible dans DMs).

  **Cas 2 : Fichiers partages "Anyone with link"**

  - Si Bob a cree des liens "anyone with link" : ils restent
    actifs après son depart
  - **Audit** : identifier ces liens
  - **Revocation** explicite

  **Cas 3 : Apps installees par Bob**

  - Apps Slack qu'il a personnellement installees peuvent
    rester
  - **Audit** : vérifier les apps de la liste blanche
  - **Suppression** des apps zombie

  **Cas 4 : Bots avec ses credentials**

  - Bots / scripts utilisant ses tokens API
  - **Audit** : identifier
  - **Migration** vers token de service

  **Cas 5 : Contenu cree par Bob**

  - Pages Notion, fichiers SharePoint, repos GitHub
  - **Restent** après son depart (en général)
  - **Audit** : identifier propriétés
  - **Transfert** ownership a un employé interne

  **Outils d'aide :**

  **Microsoft Entra ID Lifecycle Workflows** :
  - Automatise offboarding M365 + Teams + SharePoint
  - **Workflow** declenche par fin de contrat
  - **Suppression** automatique J+30

  **Slack SCIM / API** :
  - Suppression automatisee via API
  - Integration avec IAM / IdP

  **JumpCloud / Okta** :
  - **IAM unifie** : suppression dans 1 endroit propage
    partout
  - Pour PME : très efficace

  **Configuration Slack admin** :

  Members > Manage members :
  - **Filter** : Inactive
  - **Multi-channel guests** : audit reguliere

  **Configuration Teams admin** :

  Teams admin center > External access :
  - **Guests** : audit
  - **Federation** : vérifier domaines actifs

  **Le piege "il est sympa, on le laisse encore un peu" :**

  La sympathie n'est pas un critere de sécurité. Risques :
  - **Concurrent** le recrute -> info entreprise donnee
  - **Compte compromis** -> attaquant utilise
  - **Curiosite** humaine -> consultation post-depart

  Procédure systematique = pas de favoritisme.

  **Le piege "il a juste lecture, pas dangereux" :**

  Lecture = exfiltration possible :
  - **Capture écran** des conversations
  - **Telecharger** fichiers
  - **Memorisation** infos

  Lecture seule != safe. Revocation systematique.

  **Le piege "je le laisse pour qu'il finisse un dossier" :**

  Si vraiment besoin :
  - **Acces temporaire** strictement limite a UN canal
  - **Date d'expiration** stricte (max 7 jours)
  - **Documentation** justification
  - **Audit** après

  **PAS** un acces complet "au cas ou".

  **Cas reels :**

  **Cas 1 - Cabinet conseil 2024 (Slack guest oublie)**

  Consultant 6 mois ajoute a Slack workspace. Mission
  terminee. Pas de revocation. 12 mois après : audit
  decouvre :
  - Guest toujours actif
  - **45 connexions** detectees pendant 12 mois
  - **Telechargements** de fichiers : 23

  Action :
  - Revocation immediate
  - Investigation : comportement non-malveillant ?
  - Politique offboarding renforcee

  **Cas 2 - PME 2023 (Notion guest exploit)**

  Consultant fini mission. Slack revoque, mais Notion oubliee
  (workspace différent gestion).

  6 mois après : compte Notion du consultant compromis
  (mot de passe reutilise, dehashed leak). Attaquant a
  acces a Notion de la PME via le consultant.

  Cout : 30 K EUR breach données clients.

  **Cas 3 - Tesla 2018 (rappel)**

  Sous-traitant avec acces VPN + outils 6 mois après fin de
  contrat = 300 Go exfiltres. Pas une PME mais leçons
  applicables : checklist offboarding exhaustive.

  **Cas 4 - PME 2024 (succes par Lifecycle Workflows)**

  PME utilise Microsoft Entra ID Lifecycle Workflows :
  automatisation offboarding J0 + suppression J+30. Plus
  150 outils SaaS connectes (SCIM).

  Résultat : 0 compte fantome detecte sur 18 mois + audit
  ANSSI conforme.

  **Politique d'entreprise template :**

  > **Offboarding prestataires - outils collaboration**
  > 
  > **J-7** : RH/Manager informe DSI de la fin du contrat
  > avec liste des acces.
  > 
  > **J0** : Revocation IMMEDIATE (dans l'heure) :
  > - Tous outils de collaboration (Slack, Teams, Notion,
  >   SharePoint, etc.)
  > - Outils dev (GitHub, GitLab)
  > - SaaS (CRM, project management)
  > - Documentation chaque revocation
  > 
  > **J+1 a J+7** : Audit proche (logs, connexions
  > residuelles)
  > 
  > **J+30** : Audit final + documentation
  > 
  > **Tools** : Microsoft Entra Lifecycle Workflows /
  > JumpCloud / Okta automatisent.

  **La règle d'or :**

  > **Offboarding prestataire chat = checklist exhaustive +
  > revocation J0 (dans l'heure) + audit J+30. Slack, Teams,
  > Notion, SharePoint, GitHub, SaaS divers. Automatisation
  > via Lifecycle Workflows / IAM. Pas de favoritisme.
  > Documentation pour audit.**
