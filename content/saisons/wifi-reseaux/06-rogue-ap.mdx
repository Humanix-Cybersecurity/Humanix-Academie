debrief: |
  Le **Rogue AP** (Access Point clandestin) interne est un risque
  souvent sous-estime. Pas une attaque externe, mais un trou dans le
  perimetre reseau cree par un employe bien intentionne.

  **Definition Rogue AP :**

  Un access point WiFi connecte au reseau de l'entreprise SANS
  autorisation. Peut etre :
  - **Bien intentionne** : un employe qui ajoute un AP pour ameliorer
    la couverture (cas de notre scenario)
  - **Malveillant** : un attaquant qui pose un AP discret pour creer
    une porte derobee permanente
  - **Negligent** : un AP officiel mais non maintenu (ferme obsolete,
    mot de passe par defaut)

  **Risques d'un Rogue AP :**

  **1. Bypass des controles d'acces reseau**

  L'entreprise a souvent des contoles WiFi (auth 802.1X, certificats
  client, MAC filtering). Le Rogue AP les contourne : tout le monde
  qui sait le SSID/password peut entrer. Pas de log, pas de traçabilite.

  **2. Ferme/firmware obsolete**

  Un mini-routeur TP-Link consumer n'a pas de patches sur la duree
  d'un equipement enterprise. Une CVE dans le firmware = faille
  d'entree dans le reseau.

  **3. Pivot pour attaquant**

  Un attaquant externe qui prend le controle du Rogue AP (via mot de
  passe admin par defaut "admin/admin") a soudain un pied sur le
  reseau interne, contournant le firewall perimetrique.

  **4. Ressources reseau saturees**

  Un AP grand public n'est pas dimensionne pour 50 utilisateurs.
  Performance degrade, plaintes, jusqu'a impacter la prod.

  **Difference Rogue AP vs Evil Twin :**

  - **Rogue AP** : sur le reseau de l'entreprise, sans autorisation.
    Probleme INTERNE.
  - **Evil Twin** : pas sur le reseau de l'entreprise, fait croire
    qu'il l'est. Probleme EXTERNE (cf. ep 01).

  Les deux sont detectables par les memes outils (WIPS), avec des
  reponses differentes.

  **Defenses (cote DSI) :**

  **1. WIPS - Wireless Intrusion Prevention System**

  Surveillance continue du spectre WiFi par les AP officiels :
  - Detecte les SSID copies / dupliques (signature Evil Twin / Rogue)
  - Detecte les BSSID inconnus dans la plage entreprise
  - Peut faire de la "containment" (deauth packet sur l'AP suspect)

  Solutions : Cisco Meraki, Aruba, Mist Systems, Ekahau.

  **2. 802.1X (auth reseau forte)**

  Au lieu d'un simple SSID + password (PSK), chaque utilisateur
  s'authentifie avec ses identifiants AD/Azure AD + certificat
  client. Si quelqu'un copie le SSID, il ne peut pas reproduire
  l'auth 802.1X = pas de Rogue AP transparent.

  802.1X est l'equivalent WPA-Enterprise. Setup ~3 jours pour une
  PME 50 personnes. ROI eleve sur les risques de Rogue AP.

  **3. Politique IT claire**

  Charte IT signee :
  - "Aucun equipement reseau (routeur, switch, AP) ne peut etre
    branche sans autorisation"
  - "Brancher un equipement non autorise = sanction disciplinaire"
  - Sensibilisation reguliere (les juniors qui n'ont pas lu la charte)

  **4. Detection physique**

  - Inspection visuelle reguliere des espaces de travail
  - Audit physique annuel des switchs (cables qui partent vers ou ?)
  - **NAC (Network Access Control)** : tout port reseau ferme par
    defaut, ouvre uniquement apres auth 802.1X. Si le TP-Link se
    branche sur la prise reseau, le port se desactive.

  **5. Pour les cas legitimes : process de demande**

  Le junior de notre scenario a un besoin REEL (mauvaise couverture).
  La DSI doit avoir un process simple :
  - Form de demande "Couverture WiFi insuffisante zone X"
  - Audit de la zone par DSI
  - Deploiement d'un AP officiel si necessaire
  - Le junior obtient ce qu'il voulait, dans les regles

  **Cas reel - Target 2013 :**

  Le breach de Target (110M cartes bancaires volees) a commence par
  un fournisseur de climatisation qui avait acces VPN au reseau Target
  via un equipement mal securise. Pas exactement un Rogue AP, mais
  meme principe : un trou dans le perimetre cree par un acces tiers
  non audite.

  **Pour les developpeurs :**

  Si tu travailles sur des labs de test reseau / pentests :
  - **Toujours** sur reseau isole (subnet dedie, VLAN segmente)
  - **Jamais** sur le reseau de prod / dev shared
  - Documenter les equipements de test (inventaire DSI)

  Les "Pi en mode reseau" pour tester un script ? Sur un VLAN dedie
  avec accord DSI. Pas branche en RJ45 sous le bureau.

> 💡 **Pour aller plus loin** : pour la detection des Rogue AP en
> petite organisation sans budget WIPS, **Kismet** (open source) ou
> **WiFi Analyzer** sur un Android peuvent suffire pour des audits
> ponctuels. Pour la prod : Cisco Meraki Dashboard ou Mist AI sont
> les standards.
